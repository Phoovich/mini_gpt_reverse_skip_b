Reward Engineering

ออกแบบ reward function ยังไงให้ model เรียนรู้สิ่งที่เราต้องการ

ก่อนเริ่ม — จาก overview section 6:

  • Presets: (bomb→mo) = +5.30, (bomb→bmob) = −1.00

1. Sparse vs. Dense Reward

Sparse reward: ให้ reward เฉพาะเมื่อ output สมบูรณ์ถูกต้องเท่านั้น ปัญหาคือ credit assignment — model ไม่รู้ว่าก้าวไหนที่ผิด

กรณี sparse
Input: "bbbbb" — ทุก rollout ได้ reward = 0 เพราะไม่ตรง target ""
Model ไม่รู้ว่า "b เป็นปัญหา" — ทุก output ได้ signal เดียวกัน

Dense reward: ให้ partial credit ทุก rollout แม้ไม่ถูกต้องสมบูรณ์ ทำให้ model มีทิศทางในการปรับปรุง

กรณี dense (compute_reward ของเรา)
Input: "bbbbb" → target: "" (ว่าง)
Rollout ["b","b","b",""]: b_penalty = −3.0 → model รู้ทันทีว่า b คือปัญหา
Rollout [""]: len_mismatch = −0.1×|1−0| = 0; total = 0.0 (ดีกว่ามี b)

โมเดลอาจหา "shortcuts" ที่ได้ reward สูงโดยไม่ทำตาม intent ที่แท้จริง:

ที่ reward = −0.2 ต่ำมาก ไม่เพียงพอกระตุ้น model ให้ collapse KL penalty ป้องกันไม่ให้ policy เบี่ยงออกจาก SFT reference มาก

4. Term Interactions

Terms บางคู่ interact กัน:

  • exact_match + b_penalty: reinforce กัน — output ที่ถูกต้องสมบูรณ์ไม่มี b อยู่แล้ว ดังนั้น exact_match reward (5.0) ไม่โดน b_penalty หัก; output ที่มี b ไม่มีทาง exact match
  • length_mismatch + positional_match: counteract เล็กน้อย — output ที่สั้นเกินได้ len penalty แต่อาจได้ positional_match เต็มสำหรับ tokens ที่มี
  • character_coverage + b_penalty: model ที่ include b ได้ coverage credit แต่โดน b_penalty หัก → net effect คือ เอา b ออกดีกว่าเสมอ ถ้า target ไม่มี b

5. Reward Landscape — b_penalty Sweep

เมื่อปรับ b_penalty coefficient จาก −3 ถึง 0 (0 = ไม่มี penalty): reward ของ output ที่มี b กับ output ที่ไม่มี b จะ "พลิก" กัน

6. KL Coefficient Sweep

KL penalty ป้องกัน model เบี่ยงออกจาก SFT reference มากเกินไป Lagrangian form: \( \mathcal{L}(\theta) = J(\theta) - \lambda \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}}) \)

  • \(\lambda\) ต่ำเกิน → model collapse (generate <EOS> ทันที)
  • \(\lambda\) สูงเกิน → policy ขยับจาก SFT ไม่ได้ → ไม่เรียน skip b
  • \(\lambda\) = 0.2 (ค่าที่ใช้จริง) → balance ดีที่สุด

7. Code — compute_reward

mini_gpt_reverse_skip_b_rl.py:62–108

 62def compute_reward(seq, generated_tokens):
 63    target = target_skip_b(seq)
 64
 65    if "<EOS>" in generated_tokens:
 66        eos_idx = generated_tokens.index("<EOS>")
 67        pred = generated_tokens[: eos_idx + 1]
 68    else:
 69        pred = generated_tokens
 70
 71    reward = 0.0
 72
 73    # 1) exact match reward ให้แรง
 74    if pred == target:
 75        reward += 5.0
 76
 77    # 2) positional match — ถูก position และ token
 78    min_len = min(len(pred), len(target))
 79    positional_matches = sum(1 for i in range(min_len) if pred[i] == target[i])
 80    reward += 0.2 * positional_matches / max(len(target), 1)
 81
 82    # 3) character coverage
 83    pred_chars = Counter(t for t in pred if t not in ("<EOS>",))
 84    target_chars = Counter(t for t in target if t not in ("<EOS>",))
 85    coverage = sum((pred_chars & target_chars).values())
 86    reward += 0.1 * coverage / max(len(target) - 1, 1)
 87
 88    # 4) ลงโทษตัว b แรง
 89    num_b = pred.count("b")
 90    reward -= 1.0 * num_b
 91
 92    # 5) ความยาวผิดลงโทษ
 93    reward -= 0.1 * abs(len(pred) - len(target))
 94
 95    # 6) ถ้าไม่มี EOS ลงโทษเพิ่ม
 96    if "<EOS>" not in generated_tokens:
 97        reward -= 0.5
 98
 99    # 7) ลงโทษ PAD tokens
100    num_pad = pred.count("<PAD>")
101    reward -= 2.0 * num_pad
102
103    # 8) ลงโทษ special tokens ใน output
104    num_sep = sum(1 for t in pred if t == "<SEP>")
105    num_bos = sum(1 for t in pred if t == "<BOS>")
106    reward -= 2.0 * (num_sep + num_bos)
107
108    return reward
ทำไมแต่ละ term
Term 1 (+5.0): ให้ signal แรงสำหรับ perfect output — ทำให้ gradient update ใหญ่
Term 2 (+0.2): partial credit สำหรับ correct order — model รู้ว่า position สำคัญ
Term 3 (+0.1): partial credit สำหรับ correct characters แม้ผิด order
Term 4 (−1.0): core b constraint — ถ้าไม่มี model ไม่มีเหตุผลต้อง skip b
Term 5 (−0.1): ป้องกัน output สั้นมากเพื่อหลีกเลี่ยง b
Term 6 (−0.5): ป้องกัน generate ไม่จบ (ไม่มี EOS)
Term 7 (−2.0): hard penalty สำหรับ PAD ใน output (ไม่ควรเกิด)
Term 8 (−2.0): hard penalty สำหรับ SEP/BOS ใน output (ไม่ควรเกิด)

8. Stop & Check

ทดสอบความเข้าใจ

Q1: output ["<EOS>"] ทันที (input="bomb", target=["m","o","<EOS>"]) reward = เท่าไร?

เฉลย
exact=0, pos=0, cov=0, b=0,
len = −0.1 × |1 − 3| = −0.2, eos=0 (มี EOS อยู่แล้ว), pad=0, special=0
Total = −0.2

Q2: term ไหนป้องกัน model จาก generating <PAD> tokens?

เฉลย
Term 7: pad_penalty = −2.0 × count(<PAD> in pred)
เป็น hard penalty เพราะ −2.0 >> b_penalty (−1.0)

Q3: ถ้าตั้ง b_penalty = 0.0 (ลบ term 4 ออก) model จะ converge ไปที่ exact_match=100% ได้ไหม?

เฉลย
ไม่ได้: model ไม่มีแรงจูงใจ skip b → จะ converge เป็น reversed output ปกติ (เหมือน SFT)
exact_match จะต่ำ เพราะ target = reversed skip-b แต่ output = reversed with b
(สำหรับ sequence ที่ไม่มี b เลย อาจ exact match ได้ — ประมาณ 30%)
